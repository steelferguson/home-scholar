# Home Scholar — Visual Lessons Plan

## Goal

Add a **visual lesson** capability alongside the existing audio lessons, serving two very different audiences:

1. **ML/AI interview prep (adult)** — interactive visual walkthroughs of concepts like
   log/loss functions, tokenization, attention, gradient descent. Companion to the
   existing "Staff ML Interview Prep" audio content. No gamification needed.
2. **Spanish for kids (ages 6–8)** — a game-shaped experience: short Duolingo-style
   question rounds (5–10 min), with a 2D arcade game round (generic platformer /
   runner — not actual Mario IP) as the reward loop.

Both are powered by the same foundation: a new lesson content type rendered by a
step-based visual lesson engine, with content authored as JSON + a registry of
interactive React widgets.

## Key design decisions

### 1. Interactive "explorables" instead of rendered video

Rendered videos (Manim/Remotion MP4s) are slow to produce, hard to revise, and
passive. Instead, visual lessons are **step-through interactive slides**:

- Each lesson is a sequence of **steps** (like slides, but scrolly/tappable).
- A step mixes markdown text, images, math (KaTeX), and **interactive widgets**
  (e.g. a log-loss curve with a draggable predicted-probability slider, or a
  tokenizer playground where typed text splits into highlighted tokens live).
- Interactivity is better for learning and *far* cheaper to iterate on than video.
- If real videos are ever wanted later, a `video` step type can embed an MP4 from
  Supabase storage — the schema supports it from day one, we just don't build a
  video pipeline now.

### 2. Content = JSON in Supabase Storage; behavior = widget registry in the app

Follows the existing pattern (MP3s + vocab MD in storage, generated by outside
scripts):

- A visual lesson is a single `lesson_XX.json` file in a new `visual` storage
  bucket, uploaded by a script just like `upload_audio.py`.
- The JSON never contains code. Interactive pieces are referenced by id:
  `{ "type": "widget", "widget": "tokenizer-playground", "props": {...} }`.
- The app ships a **widget registry** mapping ids → React components. New concept
  visualizations = new widgets in the registry (a code change), while lesson
  text/structure/ordering is pure content (no code change).

### 3. One engine, two skins

The kids' quiz game is the same engine with different step types (question steps
instead of explainer steps), a playful theme, and a reward loop bolted on. The
arcade game is a separate full-screen React component gated by quiz performance.

## Data model changes

```sql
-- lessons: new columns
ALTER TABLE lessons ADD COLUMN lesson_type text NOT NULL DEFAULT 'audio'
  CHECK (lesson_type IN ('audio', 'visual', 'quiz_game'));
ALTER TABLE lessons ADD COLUMN content_url text;  -- JSON for visual/quiz lessons

-- courses: audience flag drives theme + dashboard grouping
ALTER TABLE courses ADD COLUMN audience text NOT NULL DEFAULT 'adult'
  CHECK (audience IN ('adult', 'kids'));

-- user_progress: reuse last_position_seconds as "last step index" for visual
-- lessons (no new column needed; the player interprets it by lesson_type).

-- NEW: quiz results + reward currency for the kids loop
CREATE TABLE quiz_results (
  id uuid DEFAULT gen_random_uuid() PRIMARY KEY,
  user_id uuid REFERENCES auth.users(id) ON DELETE CASCADE NOT NULL,
  lesson_id uuid REFERENCES lessons(id) ON DELETE CASCADE NOT NULL,
  correct_count int NOT NULL,
  total_count int NOT NULL,
  coins_earned int NOT NULL DEFAULT 0,
  completed_at timestamptz DEFAULT now()
);

CREATE TABLE player_state (
  user_id uuid REFERENCES auth.users(id) ON DELETE CASCADE PRIMARY KEY,
  coins int NOT NULL DEFAULT 0,
  streak_days int NOT NULL DEFAULT 0,
  last_played_at timestamptz
);
```

RLS mirrors `user_progress` (own rows only). Kid profiles under a parent account
are deferred (see Phase 4 options) — v1 assumes the kid uses their own login or
the parent's.

## Visual lesson JSON schema (v1)

```jsonc
{
  "schema_version": 1,
  "title": "How Tokenization Works",
  "steps": [
    { "type": "text",   "md": "## Why models can't read\nLLMs see numbers, not letters..." },
    { "type": "image",  "src": "https://.../diagram.png", "caption": "..." },
    { "type": "math",   "tex": "\\mathcal{L} = -\\log p(y)" },
    { "type": "widget", "widget": "tokenizer-playground",
      "props": { "sampleText": "unbelievable" } },
    { "type": "quiz",   "question": "Roughly how many tokens is 'unbelievable'?",
      "choices": ["1", "3", "12"], "answer": 1,
      "explain": "Sub-word tokenizers split rare words into pieces: un-believ-able." },
    { "type": "video",  "src": "https://.../clip.mp4" }   // supported, unused in v1
  ]
}
```

Steps can also be grouped into sections for a table of contents. Every step type
is a small React component; unknown types render a graceful fallback so old app
versions don't crash on new content.

## Widget registry — initial ML/AI set

Each is a self-contained SVG/canvas React component, animated with Motion
(framer-motion), math via KaTeX:

| Widget id | What it teaches |
|---|---|
| `log-explorer` | Why log? Drag x on a log curve; see log-loss blow up as p(correct)→0 |
| `tokenizer-playground` | Type text → live BPE-style token splits with colored chips |
| `softmax-slider` | Drag logits, watch probabilities renormalize + temperature knob |
| `gradient-descent-ball` | Ball rolling down a loss surface, learning-rate slider (overshoot!) |
| `embedding-space` | 2D scatter of words; similar words cluster; cosine-similarity ruler |
| `attention-heatmap` | Sentence × sentence grid; hover a word to see what it attends to |

(Build 2–3 in Phase 1, rest in Phase 2. Tokenizer uses a small bundled BPE vocab —
no API calls needed.)

## Kids' Spanish quiz-game design

**Session loop (the core ask):**

1. Pick today's lesson → **question round**: 10–15 questions, ~5–10 min.
2. Question types (all big-buttons, minimal reading, audio-first since 6–8 year
   olds read poorly): tap-the-picture ("¿Dónde está el gato?"), listen-and-choose,
   match pairs (word ↔ picture), word-order builder with word tiles.
3. Every correct answer earns **coins** + a happy animation/sound; wrong answers
   get a gentle retry (no lives, no shame at this age).
4. Round complete → **arcade time**: coins buy a round of the 2D game
   (e.g. 10 coins = one 3-minute run). Vocabulary sneaks into the game itself:
   coins in the level are labeled with the lesson's Spanish words, the door at
   the end asks one question to open.

**Audio:** question prompts need spoken Spanish. Reuse the existing TTS pipeline
(same one that makes the Pimsleur-style MP3s) to pre-generate one short MP3 per
question/word, uploaded alongside the lesson JSON. No runtime TTS calls.

**Arcade game tech:** [Kaplay](https://kaplayjs.com) (successor of Kaboom.js) —
tiny, canvas-based, built for exactly this kind of 2D platformer, embeds cleanly
in a React component. Phaser is the fallback if we outgrow it. Art: free asset
packs (Kenney.nl, CC0) for a generic jungle-runner look.

**Question bank:** generated the same way lesson scripts are today — a script in
the content projects emits `lesson_XX.json` with questions drawn from the
spanish-1 vocab lists, plus per-question audio.

## Routes & components

```
src/
├── pages/
│   ├── Lesson.jsx            # becomes a switch on lesson_type
│   ├── VisualLesson.jsx      # step player: progress dots, prev/next, keyboard nav
│   └── QuizGame.jsx          # kids mode: question round → results → arcade
├── components/
│   ├── visual/
│   │   ├── StepRenderer.jsx  # step type → component dispatch
│   │   ├── widgets/          # registry + one file per widget
│   │   └── QuizStep.jsx
│   └── game/
│       ├── QuestionRound.jsx # question types, coins, sounds
│       ├── ArcadeGame.jsx    # Kaplay canvas wrapper
│       └── RewardBar.jsx     # coins/streak header
```

`/course/:slug/lesson/:number` stays the single lesson URL; the page renders the
right player by `lesson_type`. Kids courses get a distinct theme (bigger type,
brighter palette) keyed off `courses.audience`.

New deps: `motion` (animations), `katex` + `react-katex` (math), `kaplay`
(arcade), `canvas-confetti` (celebrations). All small, no backend changes beyond
Supabase tables/bucket.

## Phased build order

**Phase 1 — Visual lesson engine (foundation)** ✅ *unblocks everything*
1. DB migration (lesson_type, content_url, audience) + `visual` bucket.
2. `VisualLesson` step player: text/image/math/quiz steps, step progress saved to
   `user_progress`, mark-complete on last step.
3. Widget registry + first two widgets: `tokenizer-playground`, `log-explorer`.
4. One pilot lesson end-to-end: **"How Tokenization Works"** JSON + upload script
   (`scripts/upload_visual.py`).

**Phase 2 — ML Interview visual set (use case 1)**
1. Remaining widgets (softmax, gradient descent, embeddings, attention).
2. Author the lesson set as a new course `ml-interview-visual` (or attach visual
   lessons into the existing interview-prep course, interleaved with audio).
   Content source: the existing Staff ML Interview Prep material — pick the
   concepts that are "better served visually" (loss/log functions, tokenization,
   attention, embeddings, backprop, evaluation metrics).
3. Dashboard shows mixed audio/visual lesson icons per course.

**Phase 3 — Kids question round (use case 2, part A)**
1. `quiz_results` + `player_state` tables; coins + streak.
2. QuestionRound with the four question types, audio prompts, kid theme.
3. Spanish question-bank generator script from spanish-1 vocab + TTS audio.

**Phase 4 — Arcade reward loop (use case 2, part B)**
1. Kaplay platformer: one tileset, 3–4 short levels, lesson vocab on collectibles.
2. Coins gate: finish round → spend coins → play; timer returns to dashboard.
3. Later options: kid profiles under a parent account (profile picker à la
   Netflix, no separate logins), level progression, word-boss doors.

## Open questions

- **Where should the ML visual lessons live** — a separate course tile, or
  interleaved into the existing interview-prep course's lesson list? (Plan
  assumes separate course; interleaving works since lessons are per-type.)
- **Kid accounts**: own Supabase login per kid vs. profile-under-parent. V1: own
  login (zero schema work); revisit in Phase 4.
- The "Staff ML Interview Prep" source content lives in the local content
  projects (not this repo) — Phase 2 authoring needs that material available to
  whatever script/session writes the lesson JSONs.
